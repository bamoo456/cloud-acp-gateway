把 console 拆成
React SPA + 多-client bridge

把 1500 行的 console.html 演進成正規前後端,目標是多裝置同時連線、管理多個 agent、未來能 review agent 變更。

frontend · Vite + React + TSbackend · long-lived Node bridge no Next · no SSR1 PR · review later
§ 01 — 為什麼

三個需求,真正的 driver 是多裝置。

關鍵洞察: 「多裝置」是後端問題——現在 bridge 硬性「同時只一個 client」,手機一連電腦就被踢(4000 superseded)。React/Next 解不了它。
  • ① 管理多個 Claude agents — 已部分支援(agents.json + folder picker)。
  • ② 多裝置同時連線(電腦 + 手機) — ⚠️ 後端要重做連線模型。這才是核心。
  • ③ 未來 review agent changes — 建在既有的 tool-diff / 權限流程上。
▸ 結論

「React + Next」是手段;真正要的是 (a) 多-client 後端 + (b) 可維護的 UI。

§ 02 — 技術決定

Vite React SPA + 長駐 Node bridge。不用 Next。

層選擇為什麼
前端使用者介面 Vite + React + TS（SPA） 私有、需登入、WS 即時工具 → 沒有 SEO/首屏/server-fetch 好處,SSR 買不到東西。Vite 產靜態檔,bridge 服務(跟現在服務 console.html 同招)。
後端ACP relay 演進現有 Node bridge 核心是活的記憶體物件:WS 連線、spawn 的 claude 子進程(握 stdio)、ledger buffer——DB 存不了、要一支長駐 process。Next 的賣點這裡用不到,WS 還得 custom server。
SQLite持久化狀態 延到 Phase 3 多-client 路由是記憶體邏輯,DB 取代不了;ledger 用 per-session 檔即可。SQLite 等 review state 才值得引入(注意:node:sqlite 需 Node ≥22,pod 是 20)。
§ 03 — 多裝置模型 C

各自獨立,看同一 session 就同步。

各自獨立

Laptop 看 session A、Phone 看 session B → 互不干擾,bridge 多工處理。

同 session 同步

兩台都看 session A → 都收到串流回覆、都能送 → 像共筆。

session = 一串 ACP 對話(session/new 建立,有自己的歷史 + 一個 claude 子進程),就是 Conversations 列表裡的一筆。

§ 05 — Phase 1 多工器

bridge 從透明 relay 變成 session-aware 多工器。

點上面看 bridge 怎麼路由。兩台裝置都看 session A。
broadcast（同 session 同步） point-to-point（id 多工） 權限路由（first reply wins）
  • N 條連線並存 — 拿掉 currentConn / supersede。
  • 通知 broadcast — session/update(X) 送給所有訂閱 X 的裝置 → 同步自動成立。
  • id 多工 — 各裝置 id 都從 1 起,bridge 改寫成全域唯一,response 只回原裝置。
  • 權限路由 — 提示送給看該 session 的裝置,第一個回應轉給 agent,其餘丟棄。
  • per-session ledger — 重連帶 cursor 補幀。
§ 06 — Phase 2 前端

三層邊界:transport / store / components。

/                  後端 bridge（不變的長駐 Node service）
  src/bridge.ts    + 服務 web/dist
  web/             前端 SPA
    src/
      lib/acp.ts     ← 唯一講 WebSocket 的地方
      lib/api.ts     ← HTTP: /history /fs /healthz
      store/         ← Zustand: sessions / cwd / agent
      components/    ← App · Sidebar · Thread · Composer …

transport lib/acp.ts

持有 WS,封裝 JSON-RPC(request/notify/onNotification/權限/reconnect)。

store Zustand

依 sessionId 套用 session/update → 多裝置同步自然成立。

▸ 契約不變

WS /acp + HTTP /history /history/messages /fs 都已存在;開對話送 session/load = 向後端訂閱該 session。
